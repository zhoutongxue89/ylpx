🦞 什么是 OpenClaw

OpenClaw 是一款开源的本地化 AI 智能体网关（Self-hosted AI Agent Gateway）。 它的核心理念是：将大型语言模型（LLM）从「对话工具」升级为「执行引擎」—— 真正能替你操作文件、控制浏览器、发送邮件、写代码、管理日历的私人 AI 助手。


你通过日常使用的即时通讯工具（WhatsApp、Telegram、Discord 等）发出指令， OpenClaw 运行在你自己的设备上，调用本地工具完成任务，再把结果回传给你。 数据不出本机，全程私密。

💡
一句话定义：OpenClaw = 本地 AI 大脑 + 多渠道消息网关 + 工具执行引擎，是一个住在你电脑里的 24/7 数字员工。

🏗️ 整体架构：六层设计
↓ 消息从顶层进入，逐层处理，向下执行，结果逐层返回 ↑

Layer 1 — Channel Adapters（渠道适配层）
统一接收来自各平台的消息，屏蔽平台差异，转换为内部标准消息格式（Envelope）
WhatsApp Telegram Discord Slack iMessage Signal 飞书/钉钉
⬇
🌐
Layer 2 — Gateway 网关（消息路由与会话管理）
Node.js 守护进程，负责维护提供商连接、验证帧格式、管理 Session 会话，根据 JSON Schema 路由消息到对应 Agent
Session 管理 RPC 调度 WS API 消息鉴权 事件发射
⬇
🧠
Layer 3 — Agent Runtime（代理运行时 / 推理核心）
OpenClaw 的核心大脑，执行 Agent Loop 推理循环，管理上下文窗口，向 LLM 发送请求并解析工具调用指令
Agent Loop Context 管理 Tool Parsing Compaction
⬇
🔧
Layer 4 — Skills / Tool Executor（技能执行层）
执行 LLM 请求的具体工具调用：文件操作、浏览器控制、代码执行、API 请求等。Skills 文件夹定义每个工具的能力边界
文件系统 浏览器控制 Shell 执行 HTTP 请求 自定义 Skills
⬇
💾
Layer 5 — Memory System（记忆持久化层）
三级记忆体系：短期上下文（对话历史）、中期 Markdown 文件（AGENTS.md / SKILL.md）、长期 SQLite 数据库（向量嵌入 + KV 存储）
SQLite Markdown Files Vector Embedding KV Store
⬇
🤖
Layer 6 — LLM Provider Interface（模型接口层）
统一的大模型调用抽象层，支持 Claude、GPT、Gemini、本地 Ollama 等，返回推理结果给 Agent Runtime
Claude GPT-4o Gemini Ollama（本地） 自定义端点

🔍 Gateway 内部处理细节

入站处理（Ingress）

  • WebSocket 帧接收与 JSON Schema 校验
  • Provider 身份认证与速率限制
  • 消息去重与幂等性保证
  • Session 查找或懒加载创建

出站处理（Egress）

  • Agent 响应序列化为渠道格式
  • 重试与指数退避
  • 媒体文件上传处理
  • 消息分片（超长内容自动切割）

♾️ Agent Loop 推理引擎

Agent Loop 是 OpenClaw 最核心的运行机制。每次收到用户消息，Agent 会进入一个 持续循环，直到任务完成或 LLM 判断无需更多工具调用为止。

① 接收消息

Gateway 将标准化 Envelope 推入 Agent 队列，包含：发送者 ID、渠道标识、原始文本/媒体

② 加载记忆 & 构建 Context

从 SQLite + Markdown 文件中检索相关记忆，拼接系统 Prompt（AGENTS.md）、历史对话、工具描述，构造最终输入

③ LLM 推理

将构建好的 Context 发送给 LLM（Claude/GPT/Gemini 等），LLM 返回文本响应或 Tool Call 指令

④ 解析 Tool Call

解析 LLM 返回的 JSON 结构，提取函数名 + 参数；若无 Tool Call，则进入 ⑧ 直接回传

⑤ 执行工具

调用对应 Skill 的实现：读写文件、执行 Shell、打开浏览器、发 HTTP 请求、操作数据库…

⑥ 写入结果 & 更新记忆

将工具执行结果追加到上下文，并根据重要性决定是否持久化到 SQLite 记忆库

⑦ 判断是否继续循环

若 LLM 仍有未完成的 Tool Call 或判断任务未完成，则回到 ③ 继续推理（最多迭代 N 轮）

⑧ 回传最终响应

循环结束后，将最终文本结果通过 Gateway 发回原始渠道，用户收到完整答复

⚠️
Context Compaction（上下文压缩）：当对话历史超出 LLM Token 限制时，OpenClaw 会自动执行压缩策略——将旧对话摘要化并存入记忆，腾出 Token 窗口空间，是防止"失忆"的关键机制。

💾 三级记忆系统

OpenClaw 拥有类似人脑的三级记忆体系，让 AI Agent 在跨会话、跨任务时保持「记性」。

🔴
短期记忆（In-Context）
当前会话的完整对话历史，直接存在 LLM 的 Token 上下文窗口中。 速度最快，但容量有限（受 Token Limit 约束），会话结束后不主动持久化。
⚡ 实时 · 高速读写
🔵
中期记忆（Markdown Files）
以 Markdown 文件存储的结构化知识，包括：
· AGENTS.md — Agent 行为规范 & 系统提示词
· SKILL.md — 每个技能的使用说明
· 自定义知识文档 可被 Agent 实时读取，支持热更新。
📄 人类可读 · 热更新
🟢
长期记忆（SQLite + Vector DB）
存储在本地 SQLite 数据库中，包含：
· 向量嵌入（Embedding）— 支持语义相似度检索
· KV 键值对 — 用户偏好、长期事实记忆
· 历史任务摘要 跨会话持久保留，智能检索相关记忆注入上下文。
🗄️ 永久保留 · 语义检索

🔧 Skills 技能体系

Skills 是 OpenClaw 的「工具箱」，每个 Skill 是一个独立目录，包含一个 SKILL.md 文件（YAML 元信息 + 使用说明），告诉 Agent 这个工具能做什么、如何调用。

📂
Skills 目录结构：每个 Skill = 一个文件夹 + SKILL.md + 可选的脚本/参考文档。OpenClaw 启动时自动扫描并热加载所有 Skills，SKILL.md 变动时实时更新。
# SKILL.md 示例结构 --- name: "browser-control" description: "控制浏览器访问网页、截图、填写表单" tools: - "browser_open" - "browser_click" - "browser_screenshot" version: "1.2.0" --- ## 使用说明 浏览器控制技能允许 Agent 打开网页、点击元素、截图... ## 示例 打开 https://example.com 并截图主页...

🔑 设计思想
Skills 采用「文档即接口」设计——SKILL.md 既是人类文档，也是 LLM 的工具说明书。Agent 读取 SKILL.md 就知道工具的能力边界和调用方式，无需写复杂的函数签名绑定代码。

🔌 适配器工作原理

标准化 Envelope（信封对象）
每个适配器将平台消息转换为统一的 Envelope 格式：

{ "id": "msg_abc123", "channel": "whatsapp", "from": "+8613800138000", "text": "帮我查一下明天天气", "timestamp": 1741996800, "media": null }

多渠道并发处理

  • 每个渠道独立运行连接维护线程
  • Session 与渠道强绑定，互不干扰
  • 同一用户不同渠道可共享长期记忆
  • Gateway 统一调度，支持水平扩展
  • 渠道断线自动重连 + 消息补发

💓 Heartbeat 心跳与自主运行

Heartbeat 是 OpenClaw 实现「24/7 自主运行」的关键机制，让 Agent 不再依赖用户触发，而是能主动执行周期性任务。

Heartbeat（心跳保活）
主会话的固定轮询机制，按配置间隔（如每60秒）触发一次 Agent 检查。执行轻量巡检：待办事项、未处理消息、系统状态等。
Cron（定时任务）
精确到分钟级的计划任务，基于 Cron 表达式配置。适用于：每天晨报、每周总结、定期备份、账单提醒等。
三问反思机制
每次心跳，Agent 会自问三个问题：① 现在有什么需要处理？ ② 上次任务完成了吗？ ③ 有什么主动可以做的？确保 Agent 保持高效自主。
分级心跳设计
支持多级频率：高频心跳（关键任务监控）、低频心跳（日常巡检）、静默模式（夜间不打扰）。通过 HEARTBEAT.md 配置文件控制。

🌙 典型自主任务场景

🌅
每日晨报
每天8点自动汇总：天气、日程、邮件摘要，发到 WhatsApp
📈
价格监控
每小时检查目标商品/股票价格，达到阈值时发出提醒
💾
自动备份
每晚自动将重要文件夹打包上传到云存储
📰
信息聚合
按主题自动抓取RSS/网页内容，生成摘要推送
🔔
截止日期提醒
监控日历和待办，提前发出智能提醒

🔐 安全与隐私模型

OpenClaw 的「本地优先」架构天然具备隐私优势，但「执行真实操作」的能力也带来了相应的安全考量。

🏠
本地运行
所有数据、记忆、配置都存储在本机。无云端同步，数据不离开设备。
📜
AGENTS.md 权限边界
通过 AGENTS.md 明确定义 Agent 的行为红线，禁止特定操作（如删除系统文件）。
🔑
消息鉴权
Gateway 对每条入站消息进行 Provider 身份验证，防止未授权指令执行。
📦
沙箱执行
代码执行 Skill 在独立沙箱进程中运行，限制资源访问范围，防止逃逸。
👁️
操作审计日志
所有 Tool 调用均记录在 SQLite 日志中，支持事后审查每一步操作。
🚨
CVE 注意事项
作为高权限本地 Agent，已发现 8 个 CVE。建议定期更新版本，不在公网裸露部署。
⚠️
生产部署建议：OpenClaw 拥有操作本地系统的高级权限，社区统计已有约 42,000 个裸机实例直接暴露在公网。强烈建议：① 使用防火墙限制访问 ② 配置 AGENTS.md 权限边界 ③ 及时跟进安全更新 ④ 生产环境避免使用 root 权限运行。
🎯 设计哲学总结
OpenClaw 的底层逻辑可以用一句话总结：
「以 Gateway 为枢纽，以 Agent Loop 为心脏，以 Skills 为四肢，以 Memory 为大脑，以 Heartbeat 为血液，打造一个能真正干活的本地 AI 自主体。」

它的架构创新在于三个彻底分离： ① 渠道与推理分离（Channel Adapter 与 Agent Runtime 解耦）、 ② 能力与配置分离（Skills 文档化 + 热加载）、 ③ 记忆与计算分离（三级记忆体系独立于 LLM）—— 这三个分离让 OpenClaw 可以灵活扩展、持续演进。
